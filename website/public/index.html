Take the vendor out of your Terraform code.

Takoform is an open-source specification and a Terraform/OpenTofu provider. You describe what you need — a bucket, a queue, a database — and a conforming host picks and operates the actual backend. The same code and state work on any host.

Why

Terraform code usually names a vendor. Writing aws_s3_bucket ties the module and its state to AWS, and moving to another cloud means rewriting. Takoform resources name only what you need, so the choice of backend stays on the host side.

aws_s3_bucket Works on AWS only. Moving elsewhere means rewriting the module and migrating state by hand.
takoform_object_bucket Works on any conforming host. Moving is a host-side change, not a rewrite.

How it works

The provider's job ends at delivering your configuration to a host. A host is any platform that implements the Takoform API — anyone can run one. The host does the creating, changing, and deleting.

01

Write

Describe what you need with typed takoform_* resources. Vendor names never appear in your code.

02

Apply

On apply, the provider sends the configuration to the host over HTTP. Nothing is changed during plan.

03

The host runs it

The host chooses the backend, holds the credentials, and makes the changes. Your state keeps IDs and public outputs only — no secrets.

More are coming

Ten is the starting point, not the limit. The specification can define any kind of service in the same data-only format, and new resources are added as specification and test additions. If something you need is missing, propose it on GitHub.

Propose a resource

The provider deliberately has no settings for backends, credentials, prices, or billing. Those belong to the host.

Design principles

Definitions are data

Resource definitions and their test fixtures are plain JSON. They contain no executable code, and the verifier rejects anything that does.

Verify, then trust

Releases are signed and pinned by digest. Anything that cannot be verified is rejected instead of being accepted by default.

The host operates

Backend choice, credentials, and operations belong to the host. The provider only submits what you asked for.

Both lifecycles tested

Terraform and OpenTofu are tested separately, end to end — not assumed to behave the same.

The specification

Everything the provider and a host agree on is written down as data-only documents.

  • spec/host-api How the provider talks to a host: discovery, preview, apply, observe, delete.
  • spec/form-definition How a service type is defined as data, and how definitions are identified.
  • spec/form-package How definitions are packaged, distributed, and verified.
  • spec/interface-declaration How a service declares what it exposes at runtime. The host still owns the record and who may use it.
  • spec/trust How releases are signed, and what gets rejected.

Current status

What works today and what does not yet, as of the latest commit.

Working today

  • The schemas and an offline package verifier
  • Ten resources with create, update, delete, import, and drift detection
  • Lifecycle tests on both Terraform and OpenTofu
  • A signed release pipeline

Not yet

  • No package has been published yet
  • Not on the Terraform or OpenTofu registries yet
  • The standard definitions are still candidates
  • Host-side install and revocation handling are still to come

Terraform のコードから、
ベンダー名をなくす。

Takoform は、サービスをベンダー中立に書くためのオープンな仕様と、その Terraform/OpenTofu プロバイダです。コードに書くのはバケット・キュー・データベースといった必要なものだけ。どのバックエンドで動かすかは、対応したホストが決めて運用します。コードと state は書き換えずに移行できます。

GitHub で見る 仕様を読む
  • MIT ライセンス
  • 標準リソースは 10 種類。今後追加していきます
  • 定義はデータのみで、コードを含みません

なぜ作ったか

Terraform のコードには、ふつうベンダー名が入ります。aws_s3_bucket と書いた時点でそのモジュールと state は AWS 専用になり、別のクラウドへ移すには書き直すしかありません。Takoform のリソースは必要なものしか書かないので、どこで動かすかをあとから変えられます。

aws_s3_bucket AWS でしか使えません。移行にはモジュールの書き直しと state の移し替えが必要です。
takoform_object_bucket 対応したホストならどこでも使えます。移行先の変更はホスト側の設定だけで済みます。

しくみ

プロバイダがやるのは、書かれた内容をホストに届けるところまでです。作る・変える・消すの実作業はホストが行います。ホストとは Takoform の API を実装した実行基盤のことで、仕様に沿っていればだれでも実装できます。

01

書く

型の決まった takoform_* リソースで、必要なものを書きます。コードにベンダー名は出てきません。

02

送る

apply すると、プロバイダは書かれた内容を HTTP でホストに送ります。plan の段階では何も変更しません。

03

ホストが動かす

バックエンドの選択、認証情報の管理、実際の変更はホストの仕事です。state に残るのは ID と公開情報だけで、シークレットは含まれません。

ここで終わりではありません

仕様は、どんな種類のサービスでも同じデータ形式で定義できるようになっています。新しいリソースは、仕様と検証テストを追加するかたちで増やしていきます。足りないものがあれば GitHub で提案してください。

リソースを提案する

このプロバイダには、バックエンドの指定・認証情報・価格・課金の設定はあえてありません。それらはホスト側で管理します。

設計方針

定義はデータだけ

リソースの定義もテストデータも、ただの JSON です。実行コードは含められず、含まれていれば検証で弾かれます。

検証してから信頼する

リリースには署名が付き、内容はハッシュで固定されます。検証できないものは受け入れず、エラーにします。

運用はホストに任せる

バックエンドの選択も、認証情報も、実際の操作もホストの担当です。プロバイダは頼まれた内容を送るだけです。

両方でテストする

Terraform と OpenTofu を同じものとは見なさず、それぞれで一連の動作をテストしています。

仕様

プロバイダとホストの間の取り決めは、4 つのドキュメントにすべて書いてあります。

  • spec/host-api ホストの見つけ方と、preview / apply / observe / delete のやりとり。
  • spec/form-definition サービスの種類をデータとして定義する方法と、その識別のしかた。
  • spec/form-package 定義をパッケージにして配布・検証する方法。
  • spec/trust リリースへの署名と、何を受け入れて何を拒否するか。

いまの状態

できていることと、まだできていないことをそのまま書いています。

できていること

  • スキーマ一式と、オフラインで動くパッケージ検証ツール
  • 作成・更新・削除・import・drift 検出まで対応した 10 種類のリソース
  • Terraform / OpenTofu それぞれでのライフサイクルテスト
  • 署名付きリリースのしくみ

まだできていないこと

  • パッケージはまだ公開していません
  • Terraform / OpenTofu の Registry にはまだ載っていません
  • 標準の定義はまだ候補の段階です
  • ホスト側の取り込み・失効の処理はこれからです
